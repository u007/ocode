# How to Run an Evaluation

This is a runbook **and** a ready-to-paste AI prompt. It takes one **stack** +
one **exact model** and produces two artifacts: a scorecard and a derived skill.

## Rule 0 — CLOSED-BOOK. The answerer NEVER sees the answer key.

`questions.yaml` and `questions.md` embed the `answer` + `rubric` on **every**
record. They are the **grader's answer key** — showing them to the model being
evaluated makes it copy the reference and score a meaningless ~100%. This has
happened: an open-book run scored a flat 100% across all 200 questions, including
this repo's un-learnable house rules, because the answering agent could read the
corpus. Do not repeat it.

Two roles, **two separate agents/sessions**, strict information barrier:

| role     | may read                                             | must NOT read |
|----------|------------------------------------------------------|---------------|
| ANSWERER | **only** `docs/okf/_prompts/<stack>.md` (questions)  | `questions.yaml`, `questions.md`, `scores/`, `derived/`, `meta.yaml`, ANY repo file, the web for these Qs |
| GRADER   | `questions.yaml` (answer key) + the answerer's answers | — |

- The answer sheets in `_prompts/` are generated by
  `python3 docs/okf/_tools/gen-prompt-sheets.py` (id + question only). Regenerate
  after editing any `questions.yaml`.
- ANSWERER and GRADER must not be the same agent/session — a model that saw the
  key while "answering" then grades itself generous. If you run the answerer as a
  sub-agent, give it the `_prompts/` sheet as its ONLY input and forbid repo access.
- Answers land in `docs/okf/<stack>/answers/<model-id-flattened>.md`. The grader
  reads that file, never re-asks the answerer.

## What you need

- A stack folder under `docs/okf/<stack>/` with a `questions.yaml` (react,
  golang, rust, tanstack, nextjs, conduct today) and its generated
  `docs/okf/_prompts/<stack>.md` answer sheet.
- The **exact** model id + version you're evaluating, **provider-stripped** (e.g.
  `tencent/hy3` @ `3.0`, not `novita/tencent/hy3`). Never a family name.

## The two ways to run it

### A. Manual (you drive)

1. **Answer (closed-book).** Give the target model **only**
   `docs/okf/_prompts/<stack>.md` — the questions, no answers. Collect its replies
   into `docs/okf/<stack>/answers/<model-id-flattened>.md` as a YAML list of
   `{id, answer}`. Each question is a fresh context (testing the model, not its
   memory of your last answer). The model must not see `questions.yaml`.
2. **Grade (open-book, separate step).** For each answer, open that record's
   `rubric` in `questions.yaml` and grade per `_schema/rubric-guide.md`: award
   each `point` whose concept is genuinely present; if none but a `partial`
   matches, award the partial.
3. Fill `_schema/scorecard.template.md` → save as
   `docs/okf/<stack>/scores/<model-id-flattened>.md`.
4. Read the per-tag subscores. For every tag `< threshold` (0.75), write a
   corrective section in `docs/okf/<stack>/derived/<stack>.<model-id-flattened>.SKILL.md`.
   Cover **only** weak tags — say nothing about tags the model aced.

### B. Prompt an AI to do it (paste the block below)

The GRADER agent reads the corpus + the answerer's answer file and writes both
output files. It must **never** be the same agent that produced the answers, and
the answers must have been produced closed-book (Rule 0). Feeding the grader
answers that were written with the key visible produces a fake 100% — the grader
is only as honest as the answers are blind.

---

## Answerer prompt (closed-book — give this to the TARGET model)

Give the target model the contents of `docs/okf/_prompts/<STACK>.md` and this
instruction. Do NOT give it repo access. If it's a sub-agent, the sheet is its
ONLY input.

```
Answer the questions in the sheet below from your own knowledge ONLY. Do not
search the web, do not open any repository file, do not look for an answer key.
Answer each question independently. If you don't know, say so — a wrong guess
scores the same as "unsure" but a copied answer invalidates the whole run.
Return a YAML list of { id, answer } exactly matching the question ids.

<paste docs/okf/_prompts/<STACK>.md here>
```

Save its reply to `docs/okf/<STACK>/answers/<MODEL_ID-flattened>.md`, then run the
grader below.

## Grader prompt (open-book — separate agent/session)

```
You are the GRADER in a Kaizen evaluation of this repo. Read docs/okf/README.md
and docs/okf/_schema/{question-format,rubric-guide,scorecard.template,stack-detection}.md
first so you follow the exact format.

You grade answers that were ALREADY produced closed-book by the target model
(see Rule 0 in HOW-TO-EVALUATE.md). You do NOT answer the questions yourself —
you have the answer key, so any answer you write would be a copy. If the answer
file is missing, STOP and report it; do not substitute your own answers.

INPUTS (fill these in):
- STACK: <react | golang | rust | tanstack | nextjs | conduct>
- MODEL_ID: <PROVIDER-STRIPPED model id, e.g. claude-opus-4-8 or tencent/hy3>
  # Strip the leading provider segment: `novita/tencent/hy3` -> `tencent/hy3`,
  # `anthropic/claude-opus-4-8` -> `claude-opus-4-8`. NEVER a family name, and
  # NEVER keep the provider prefix — one eval must cover the model on any host.
- MODEL_VERSION: "<x.y>"
- EVALUATED_VIA: <host you RAN it through: novita | openrouter | anthropic | ...>
  # Provenance only, NOT part of the key. Exception: if this host serves a
  # materially different quantization (fp8/fp16/int4), it's a distinct eval.
- ANSWERS_FILE: docs/okf/<STACK>/answers/<MODEL_ID-flattened>.md
  # The closed-book answers to grade. Produced separately by the target model
  # given ONLY docs/okf/_prompts/<STACK>.md. If it doesn't exist, STOP.

STEPS:
1. Load docs/okf/<STACK>/questions.yaml (the answer key) and its meta.yaml.
   Note corpus_rev. Load ANSWERS_FILE (the closed-book answers), matched by id.
2. For each question, grade the answer from ANSWERS_FILE
   against that record's rubric per rubric-guide.md:
   - award every `point` whose concept is genuinely present (own wording is fine),
   - else award a matching `partial`,
   - normalized = min(awarded, full) / full.
3. Compute per-tag subscores = Σ(normalized×weight)/Σ(weight) over each tag's
   questions. Mark any tag with <4 questions as (low-n).
4. Write docs/okf/<STACK>/scores/<MODEL_ID>.md from scorecard.template.md.
   FILENAME: flatten "/" in MODEL_ID to "__" (e.g. tencent/hy3 -> tencent__hy3.md;
   claude-opus-4-8 unchanged). Fill front matter (model_id [provider-stripped],
   model_version, evaluated_via, evaluated_on, stack, stack_corpus_rev = the
   corpus_rev from step 1, threshold: 0.75), the per-question table, per-tag
   subscore table, stack score, and derivation targets.
5. For every tag with subscore < 0.75, write ONE corrective section in
   docs/okf/<STACK>/derived/<STACK>.<MODEL_ID-flattened>.SKILL.md. Rules:
   - Cover ONLY below-threshold tags. Say NOTHING about tags at/above threshold
     (the model already knows them — restating wastes prompt/cache budget).
   - Front matter: name, description, when_to_use (gate on: active model's
     canonical id == <MODEL_ID> AND repo uses <STACK>), plus tuned_for (= the
     provider-stripped MODEL_ID, slashes kept), tuned_version, stack,
     source_scorecard, threshold, revalidate_when: model_version changes.
   - Each section = concrete corrective guidance for that tag's real failure,
     drawn from what the model actually got wrong in the grading.
6. Do NOT overwrite the illustrative react/scores/claude-opus-4-8.md placeholder
   unless STACK=react and MODEL_ID=claude-opus-4-8 and you are replacing it with
   a REAL evaluation — in which case remove the "ILLUSTRATIVE" banner.
7. Validate: scorecard front matter is complete, every below-threshold tag has a
   skill section and no above-threshold tag does. Report the stack score, the
   per-tag table, and the tags you derived.

Do not commit unless asked.
```

---

## After the eval

A real scorecard + derived skill for one (stack × exact model) is the thing that
unblocks the enforcement hook (`TODO.md` → "Wire the enforcement hook"). Until at
least one real pair exists, the runtime injection stays unbuilt on purpose.

**Version discipline:** re-run the whole eval when the model's version changes —
a scorecard is valid only for the exact `model_id` + `model_version` it names.
