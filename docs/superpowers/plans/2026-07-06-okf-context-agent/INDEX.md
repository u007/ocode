# OKF Knowledge Bundle + Context Agent — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. Per user planning rules this plan is high-level: files and functions to touch, behaviors to test — no code snippets.

**Spec:** `docs/superpowers/specs/2026-07-06-okf-context-agent-design.md` (read it first; it is the source of truth for all format rules and invariants).

**Goal:** OKF v0.1 knowledge bundle in `docs/`, curated by a new small-model `context` subagent (sole automated writer), with index injection, `knowledge_lookup`, post-job maintenance, `task_cancel`, and `/docs init|update|cleanup` commands.

**Architecture:** New `internal/knowledge` package owns all bundle mechanics (frontmatter round-trip, index/log generation, search, cross-instance lock). Agent layer wraps it in tools injected only onto the `context` subagent at dispatch. Maintenance mirrors `internal/agent/memory_maintenance.go` (channel + worker + small-model triage) with added shutdown drain and file lock. TUI extends the existing `/docs` command.

**Tech stack:** Go, `gopkg.in/yaml.v3` (new direct dep, already in go.sum) with `yaml.Node` for unknown-key round-trip, flock-style lock file.

## Global constraints (apply to every task)

- Activation gate: knowledge system active only when `DocPromptEnabled` AND root `docs/index.md` has `okf_version: "0.1"` frontmatter marker (created only by `/docs init`). No bundle write ever without prior explicit init.
- Every bundle mutation (worker, init, cleanup) goes through `knowledge.WithBundleLock` (`docs/.okf.lock`). No write path outside it.
- `doc_write`/`doc_deprecate` refuse paths outside bundle root and refuse reserved files (`index.md`, `log.md`) — those are regenerated by the tool layer only.
- Unknown frontmatter keys MUST survive rewrite (OKF round-trip rule). `type` required on every written doc.
- Deletion only via `/docs cleanup` with per-file confirmation. Maintenance may only set `status: deprecated`.
- All caught errors logged with context (project rule); maintenance is best-effort, never blocks/fails the user session.
- Listings (search results, index entries, cleanup lists) sorted and paginated.
- Validate with `go build ./...` and `go test ./internal/...` per task; TDD (failing test first) where the unit is testable.
- Commit per task with conventional message + `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>`.
