# Plan: Scheduled jobs / cron dispatch for ocode (nanobot + Claude Code research)

**Created:** 2026-07-17 13:28:37 +08
**Status:** Draft

## Context

The user wants ocode to gain a **scheduled-job / cron dispatch** capability — "we need the
scheduled job, alike claude code dispatch." Two reference systems were studied:

1. **nanobot** (`~/www/nanobot`) — already has a working `CronService` + `cron` tool.
2. **Claude Code** — has `/loop`, Desktop scheduled tasks, and Cloud Routines, all built on
   `CronCreate` / `CronList` / `CronDelete` tools with specific dispatch semantics.

ocode currently has **no scheduling at all** (grep for `cron|schedul|ticker|time.Tick` in
`internal/` returns nothing). This plan closes that gap by adopting nanobot's clean
scheduler/service separation and Claude Code's dispatch ergonomics.

### What nanobot does (reference implementation)
- `nanobot/cron/service.py` — `CronService`: a **scheduler-only** engine.
  - Disk-backed JSON store (`cron/jobs.json`), auto-reloaded if modified externally.
  - Timer loop (`_arm_timer` → `tick` → `_on_timer` → `_execute_job`) that polls in short
    legs capped at `_MAX_POLL_S = 60s` (drift protection on throttled VMs).
  - Schedule kinds: `at` (one-time ISO), `every` (interval seconds), `cron` (5-field expr +
    IANA `tz` via `croniter`).
  - Fires by calling an injected `on_job: Callable[[CronJob], Coroutine]` callback.
  - Recomputes next-run on start, preserves future run times across restarts.
  - One-shot (`at`) jobs auto-delete after running.
  - Ownership model: per-user or `team:`-prefixed; `owned_job` decorator enforces access.
- `nanobot/agent/tools/cron.py` — the `cron` tool (actions: `add`, `list`, `remove`,
  `describe`) the LLM calls to manage jobs. Scopes each job to `channel`/`chat_id`.
- `nanobot/cli/commands.py:462` `on_cron_job` — the dispatch glue:
  ```python
  response = await agent.process_direct(
      message, session_key=f"cron:{job.id}",
      channel=..., chat_id=..., user_id=job.owner or None)
  if job.payload.deliver and job.payload.to:
      await bus.publish_outbound(OutboundMessage(...))  # deliver result to channel
  ```
  i.e. it builds a **"Scheduled job context"** prefix and runs a fresh agent turn in a
  dedicated session, then optionally delivers the result back to the chat.

### What Claude Code does (dispatch ergonomics to copy)
- **Three tiers:** `/loop` (session-scoped, needs open session, ≥1min, 7-day expiry, Esc to
  stop, built-in maintenance prompt via `loop.md`); **Desktop scheduled tasks** (your machine,
  no open session, ≥1min, per-task permissions, local files/MCP); **Cloud Routines** (`/schedule`,
  Anthropic-managed infra, no machine needed, ≥1hr, fully autonomous, connectors, fresh clone).
- **Tool surface:** `CronCreate` / `CronList` / `CronDelete`; each task gets an 8-char ID;
  a session holds up to 50 tasks.
- **Dispatch semantics (the important part):**
  - Scheduler checks every second for due tasks, enqueues at **low priority**.
  - A scheduled prompt **fires between turns, never mid-response**; if Claude is busy it waits
    for the current turn to end.
  - All times interpreted in the **local timezone**.
  - **Jitter** to avoid every session hitting the API simultaneously.
  - **7-day expiry** for recurring tasks; **one-shots delete after firing**.
  - Natural-language authoring ("remind me at 3pm…", "schedule daily PR review at 9am").
  - Skill-invocation guardrails: a scheduled fire only runs skills Claude is allowed to invoke
    on its own (`disable-model-invocation`, `skillOverrides`, deny rules).

## Goals

1. Add a persistent, disk-backed **scheduler engine** to ocode (modeled on nanobot's
   `CronService`): `at` / `every` / `cron` schedules, timezone support, restart survival.
2. Expose it to the agent via a **`cron` tool** (`add`/`list`/`remove`/`describe`) and a
   **`/cron` slash command** (natural-language authoring + list/cancel), mirroring
   Claude Code's `CronCreate/List/Delete` ergonomics.
3. Wire a **dispatch glue** so a fired job runs a real ocode agent turn
   (`agent.Step`) with a "Scheduled job context" prefix, in a dedicated session, and
   optionally delivers the result (to the session transcript / a configured channel).
4. Honor Claude Code dispatch semantics: low-priority enqueue between turns, jitter, 7-day
   expiry, one-shot auto-delete, local timezone, max-jobs cap.
5. Integrate with ocode's **permission system** so autonomous jobs are safe by default
   (no silent destructive actions unless explicitly opted in).
6. Ship with unit tests for the scheduler core + an integration test that fires a job, plus
   docs.

## Approach

Adopt nanobot's separation of concerns: a scheduler **service** (timing + persistence +
ownership) is dumb about the agent; a thin **dispatch callback** (set where the agent lives)
turns a due job into an agent turn. The agent entry point already exists and is proven in the
server path: `agent.NewAgent(client, tools, cfg, lspMgr)` + `ag.Step(messages)`
(`internal/server/handler.go:231,468,616`; `internal/server/handler_sse.go:174,221`), and
`h.getOrCreateAgentSession(id)` manages per-session agents. So the dispatcher reuses the exact
machinery the web server uses.

### Package layout (new)
- `internal/scheduler/scheduler.go` — `Service`: timer loop (`armTimer`/`tick`/`onTimer`/
  `executeJob`), `AddJob`/`ListJobs`/`RemoveJob`/`DescribeJobs`, `Start`/`Stop`, `onJob`
  callback, disk load/save with external-modification reload (croniter-equivalent via
  `github.com/robfig/cron/v3` for 5-field + optional `* * * * *` + location/timezone).
- `internal/scheduler/types.go` — `Job`, `Schedule{kind,atMs,everyMs,expr,tz}`,
  `Payload{kind,message,notes,owner,deliverTo}`, `JobState{nextRunAtMs,lastRunAtMs,
  lastStatus,lastError}`, `Store{version,jobs}`.
- `internal/scheduler/scheduler_test.go` — compute-next-run, persistence round-trip,
  external-reload, 7-day expiry, one-shot delete, jitter, drift-poll.
- `internal/tool/cron.go` — the `cron` tool (`Tool` interface impl) registered in
  `internal/tool/tool.go:InitBuiltinTools` (single source of truth per AGENTS.md).
- `internal/scheduler/dispatch.go` — `OnJob` glue: build the context prefix and call
  `agent.Step` via the host's agent-session factory; deliver result.

### Scheduler lifecycle / where it runs (advisor-verified)
The scheduler must live in a **long-lived process** so jobs fire without an open TUI session
(matches Claude Code Desktop tasks / nanobot gateway). Verified hosts:
- `serve`/`web` subcommands go through `server.Run` (`main.go:96-120`) — **primary host**.
  (Note: the primary `ocode` binary is `main.go`; `ocode` with no subcommand starts the TUI at
  `main.go:167-195, 220-229`. There is **no `cmd/ocode`** despite AGENTS.md — `cmd/` only has
  `ocode-desktop` and `ocode-telegram`.)
- The desktop app boots the HTTP/SSE server in a background goroutine
  (`cmd/ocode-desktop/main.go:58-66`, `internal/desktop/boot.go:23-56`) — secondary host.
The **TUI does not host the scheduler** (it is ephemeral); it only authors/manages jobs via the
`cron` tool and `/cron` command, which write to the shared on-disk store. On server/desktop
start, `Service.Start()` loads the store and arms the timer. On stop, `Service.Stop()` cancels
the timer task. (When the user only runs the TUI with no server/desktop up, jobs simply don't
fire until a long-lived host is running — document this like Claude Code's "Desktop tasks need
your machine on".)

### Dispatch flow (per fired job)
1. `Service.onTimer` finds due jobs → `executeJob(job)`.
2. `onJob(job)` callback (set by the host) builds:
   ```
   [Scheduled job context]
   Job: <name>
   Purpose: <notes>
   Scheduled by: <owner>
   Created: <UTC time>
   Schedule: <human cron summary>
   ---
   <job.payload.message>
   ```
3. Dispatch uses a **headless runner built around `agent.Step`** (the same entry the server
   SSE and TUI stream paths end at — `internal/server/handler_sse.go:198-229`,
   `internal/tui/model.go:11611`). The TUI's `streamStep` extras are UI-only plumbing; do NOT
   call `streamStep`. Construct an `agent.Agent` via `agent.NewAgent(client, tools, cfg, lspMgr)`
   — `NewAgent` automatically starts compaction/memory maintenance workers
   (`internal/agent/agent.go:583-666`, `CompactAsync`/`MaybeCompactAsync` at `:1254-1318`), so a
   long-lived per-job agent gets those for free.
4. **Persistent per-job session** `cron:<id>` (advisor-verified safer than a fresh ephemeral
   session each fire: `session.Save`/`Load` at `internal/session/session.go:126-235`; `NewAgent`
   has no public shutdown for its workers, so "one long-lived job agent/session" is the right
   shape — mirrors nanobot's `cron:{id}` session key). Reuse a `getOrCreateAgentSession`-style
   factory keyed by `cron:<id>`.
5. **Retention / compaction policy (must-have, not optional):** `session.Save` overwrites the
   full transcript and **never prunes** (`internal/session/session.go:126`). Add a policy so a
   recurring job's `cron:<id>` session is compacted or capped (e.g. trim to last N turns / a
   token ceiling, or call `MaybeCompactAsync` periodically) to avoid unbounded growth.
6. If `deliverTo` is set, push the result to that target (session transcript by default;
   Telegram/RC path later). On success set `lastStatus=ok`; on panic/error set `lastStatus=error`
   and keep the job (so it can be retried next cycle) unless one-shot.

### Permissions (safety-first, modeled on ocode-permissions skill) (advisor-verified)
- Each job records an `owner` and a **permission mode**: `ask`/`normal` (default — deny/ask on
  destructive tools, like normal sessions) or `auto` (YOLO-style auto-allow, explicit opt-in
  only). Cloud-routine-style "fully autonomous" is the `auto` mode and must be an explicit
  choice, never the default.
- **How to bind it (corrected):** there is **no exported `Agent.SetPermissions`**. The
  permission policy is the private `permissions *PermissionManager`, set from config inside
  `NewAgent` (`internal/agent/agent.go:583-666`); `Agent.SetMode` is *workflow* mode
  (build/plan/review…), **not** permission mode (`internal/agent/agent.go:1734-1758`,
  `internal/agent/mode.go:10-18`). Runtime permission-mode control lives on
  `PermissionManager.SetMode` (`internal/agent/permissions.go:2219-2242`). Cleanest approaches:
  (a) **clone `cfg`** and set the permission mode on the cloned config *before* calling
  `NewAgent` for the job's agent, or (b) add a small **forwarding setter** on `Agent` that wraps
  `permissions.SetMode`. Prefer (a) to avoid touching the agent internals; this reuses the
  existing permission layer rather than bypassing it.
- Skill-invocation guardrails: a scheduled fire only invokes skills the user has marked
  model-invocable (mirror Claude Code's `disable-model-invocation` rule) — gate in the cron
  tool/`OnJob`.

### Schedule authoring UX (Claude Code-style)
- `cron(action="add", message, name?, notes?, every_seconds? | cron_expr? (5-field) + tz? | at? (RFC3339), deliver_to?)`.
- `/cron` natural-language parser: "remind me at 3pm to …", "schedule daily PR review at 9am",
  "`/cron list`", "`/cron cancel <id>`", "`/cron describe`". Parse intervals like `5m/2h/1d`
  (seconds rounded up to 1-min granularity, like Claude Code).
- `every_seconds` and `cron_expr` maps to `every` / `cron` kinds; `at` maps to one-shot `at`.
- 8-char job IDs (like Claude Code); per-project cap (default 50, configurable).

### Persistence location
Store under the project-scoped global data dir, e.g.
`GlobalDataDir()/scheduler/<project-slug>/jobs.json`
(`internal/paths.GlobalDataDir()` returns `~/.local/share/opencode` on macOS). This survives
restarts and is consistent with ocode's per-project session layout.

## Implementation Steps (advisor-recommended build order)

1. **Add dependency** — `github.com/robfig/cron/v3` (not currently in the module) for 5-field
   cron + IANA location/timezone; use it in `_computeNextRun` for `cron` schedules.
2. **Scheduler store + types + tests (`internal/scheduler`)** — `types.go` (Job/Schedule/
   Payload/Store) and `scheduler.go` (`Service` with `Start`/`Stop`, `AddJob`/`ListJobs`/
   `RemoveJob`/`DescribeJobs`, disk load/save with mtime-based external reload, `_computeNextRun`
   via robfig/cron for `cron` + simple math for `at`/`every`, `_armTimer` with 60s poll legs,
   `_onTimer` → `_executeJob` calling `onJob` callback, recompute-next-run-on-start, one-shot
   auto-delete, 7-day expiry sweep, jitter offset). Add `scheduler_test.go`. **Add a file lock
   around store read/write** (see Risks — `session.Save` has no lock and TUI+server+desktop can
   race on the shared store).
3. **Shared headless dispatch helper (`internal/scheduler/dispatch.go`)** — `OnJob(job)` builds
   the context prefix, obtains a **persistent per-job `cron:<id>` session** via a
   `getOrCreateAgentSession`-style factory, constructs the agent via `agent.NewAgent(...)`,
   calls `ag.Step(messages)`, enforces the retention/compaction policy for the session, delivers
   the result, records `lastStatus`/`lastError`. Wrap in panic recovery so a bad job can't kill
   the host loop. This is the de-risking core — build and test it before any UI.
4. **Cron host wiring (server + desktop)** — in `server.Run` (`main.go:96-120`) construct the
   `Service` with the store path + `onJob` glue, `Start()` after the server is up, `Stop()` on
   shutdown; do the same in `cmd/ocode-desktop/main.go` (the desktop already boots the server in
   a bg goroutine). TUI (`main.go` no-subcommand) only authors/manages jobs.
5. **Permissions + job API** — per-job permission mode via cloned `cfg` (or a forwarding setter
   on `Agent`) before `NewAgent`; enforce skill-invocation guardrails. Expose a small
   `POST /api/cron` + `GET /api/cron` for the web UI (optional, mirrors `/api/agents/runs`).
6. **`cron` tool + `/cron` slash command** — implement the `Tool` (`add`/`list`/`remove`/
   `describe`) and register in `InitBuiltinTools` (keep deterministic tool ordering per
   AGENTS.md prompt-cache rules). Add the `/cron` natural-language command as an **instant**
   command. Default permission: allow `list`/`describe`/`remove`; require review for `add`.
7. **Delivery / result surfacing** — decide and implement how a fired job's result reaches the
   user: append to the `cron:<id>` session transcript (default), push via SSE notification,
   and/or a `deliverTo` hook for Telegram/RC later.
8. **Docs** — `docs/scheduled-jobs.md` concept doc + README/usage updates; clearly state the
   "TUI authors, server/desktop fires" model and the "jobs don't fire unless a long-lived host
   is running" caveat.

## Risks & Considerations

- **Which process hosts the scheduler?** Must be server/desktop, not the TUI (TUI is
  ephemeral). If the user only runs the TUI, jobs won't fire until a server/desktop instance is
  up — document this clearly (matches Claude Code "Desktop tasks need your machine on").
- **Restart survival / drift** — reuse nanobot's short poll legs (`_MAX_POLL_S`) and
  recompute-next-run-on-start so jobs aren't skipped after a pause; preserve future run times.
- **Store file-lock (advisor blind spot)** — `session.Save` has **no file lock**
  (`internal/session/session.go:126`), and TUI + server + desktop can all read/write the shared
  scheduler store concurrently. Wrap store read/write in a `sync.Mutex` (or an `flock` like the
  knowledge bundle's `WithBundleLock`) so external edits from the TUI (e.g. `/cron remove`) and a
  firing dispatch don't race. Reuse nanobot's mtime-reload trick so external edits are picked up.
- **Per-job session growth (advisor blind spot)** — `session.Save` overwrites the full transcript
  and never prunes. A recurring job's persistent `cron:<id>` session grows unbounded unless a
  retention/compaction policy is enforced (trim to last N turns / token ceiling, or call
  `MaybeCompactAsync`). This is a must-have, not optional.
- **Cron dependency (advisor blind spot)** — `github.com/robfig/cron/v3` is **not** yet a module
  dependency; add it in step 1. Validate it supports 5-field expressions + `cron.WithLocation`
  for timezone.
- **Result surfacing (advisor blind spot)** — decide upfront how a fired job's result reaches the
  user: session append (default), SSE notification, and/or a `deliverTo` hook (Telegram/RC). Don't
  leave it unspecified — an autonomous job whose output goes nowhere is useless.
- **Prompt-cache tool ordering (advisor blind spot)** — registering the `cron` tool changes the
  tools array and busts the Anthropic prompt cache (AGENTS.md: tools come first in the prefix).
  Keep `InitBuiltinTools` deterministic (it already sorts), and treat the tool set as
  grow-only/sticky within a session so only the first attach pays the re-cache.
- **Concurrency** — a job firing must never block the live agent turn; run dispatch in a
  goroutine at low priority. The store lock + mtime-reload covers cross-process mutation.
- **Permission safety** — default `ask`/deny on destructive tools; `auto` mode is explicit opt-
  in only. Never let a scheduled job bypass the permission layer (use cloned `cfg`, not a bypass).
- **Timezone** — all schedules in the host's local zone unless `tz` given (Claude Code default);
  store `tz` explicitly so results are reproducible.
- **Cap & expiry** — enforce max-jobs (default 50) and 7-day recurring expiry to bound resource
  use; one-shots auto-delete.
- **Panic isolation** — `executeJob` must recover per-job panics so one bad job can't take down
  the host loop (ocode has known "no panic recovery in Update/View" gaps; the scheduler must be
  self-contained and panic-safe).
- **Multi-frontend** — TUI, server, desktop, telegram all read the same store; ensure the
  on-disk format is the single source of truth and the timer is armed exactly once per host.

## Validation

- **Unit** (`go test ./internal/scheduler/...`):
  - `_computeNextRun` for `at`/`every`/`cron` (incl. timezone via `robfig/cron` location).
  - Store save/load round-trip + external-modification reload.
  - One-shot deletes after run; recurring reschedules; 7-day expiry sweep.
  - Jitter offset applied; poll-leg cap respected (no single multi-hour sleep).
- **Integration** (new `internal/scheduler` test or server test): inject a fake `onJob` that
  records the call; advance the clock / use a 1s `every` job; assert the callback fires with the
  correct context prefix; assert `lastStatus=ok`.
- **Tool contract** (`go test ./internal/tool/...` or a focused test): `cron` add/list/remove/
  describe return expected shapes; invalid schedule rejected.
- **Manual**: run `ocode` in serve mode, issue `/cron add every_seconds=60 message="say hi"`,
  confirm a turn fires within ~60s and the result appears in the session; `/cron list` shows it;
  `/cron cancel <id>` stops it; restart the server and confirm the job reloads and fires again.
- **Build/lint**: `go build ./...`, `go vet ./...`, `gofmt` on new files.
