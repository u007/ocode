# Plan: Add Novita live model fetch with pricing and context

**Created:** 2026-07-07
**Status:** Draft

## Context
The Novita AI provider is registered in auth and client config, but models come only from the static models.dev snapshot. Users want to see Novita's live model list (with pricing and context size) when they have credentials configured, similar to LM Studio and Requesty live fetches.

## Goals
1. Fetch live model list from Novita's API when credentials are available
2. Include context window size and pricing data from the API response
3. Fall back to the models.dev snapshot if the live fetch fails
4. Ensure `ModelCost` can resolve pricing for live-fetched models

## Approach
Follow the established LM Studio/Requesty pattern:
1. Add `fetchNovitaLiveModels()` — calls `GET /v1/models` on Novita's base URL, parses the rich response including `context_size`, `input_token_price_per_m`, `output_token_price_per_m`, features
2. Add a package-level cache (`novitaLiveData`) for the fetched model entries so `ModelCost` can access pricing
3. Skip `novita-ai` in snapshot-based model listing; use live fetch with snapshot fallback
4. Handle `novita-ai` in `providerModelsFromRegistry()` for individual provider lookups
5. Modify `modelEntryFor()` to check the Novita live cache as an additional data source

## Files to Change
- `internal/agent/models_registry.go` — add live fetch, cache, and integrate into all model listing/lookup paths

## Implementation Steps
1. Define response types and add `fetchNovitaLiveModels()` with API call
2. Skip novita-ai in `allProviderModelsFromRegistry()`, add live models with snapshot fallback
3. Handle novita-ai in `providerModelsFromRegistry()` 
4. Check novita live cache in `modelEntryFor()` so `ModelCost` resolves pricing

## Risks & Considerations
- Pricing unit conversion: Novita API returns `input_token_price_per_m` in 1/10000th of a dollar (divide by 10000 to get USD/M)
- API call must only fire when credentials are available (env var or stored key)
- Must handle API errors gracefully and fall back to snapshot

# Plan: Sub-agent goroutine/memory leak — synchronous (done) + background (gap fix)

**Created:** 2026-07-07 (appended)
**Status:** B1 implemented & validated; background gap fix pending approval

## Context

Every `knowledge_lookup` (and any synchronous `task` dispatch) builds a fresh
sub-`Agent` via `NewAgent`. `NewAgent` spawns two goroutines that were never
stopped on the synchronous path:

- `memoryMaintenanceWorker` loops `for req := range a.memoryMaintCh`.
  `memoryMaintCh` was **never closed anywhere** in the codebase, so the
  goroutine blocks forever.
- `docMaintenanceWorker` only exits when `docMaintCh` is closed
  (`docMaintShutdown`), but the synchronous path never called `Shutdown`.

Result: each synchronous dispatch leaked one `Agent` + 2 blocked goroutines +
its `snapshotStore`/`tools` map permanently (they stay reachable via the
blocked goroutines). Background `task` dispatches have the same leak — the
run record (`run.Sub`) keeps the `Agent` referenced after completion.

(Investigation also confirmed `QueueMemoryMaintenance`/`QueueDocMaintenance`
have no callers on sub-agents, so a sub-agent's maintenance channels are always
empty at teardown — closing them drops no in-flight work.)

## Status: B1 — synchronous teardown (DONE, validated)

Changes already applied:
- `internal/agent/agent.go`
  - Added `memoryMaintDone chan struct{}` + `memoryMaintShutdownOnce
    sync.Once` fields; initialize `memoryMaintDone` in `NewAgent`.
  - Added `shutdownTransient()`: calls `Cancel()` (closes `stopCh`) +
    `docMaintShutdown()` + `memoryMaintShutdown()`, **without** resetting the
    snapshot store. This deliberately avoids `Shutdown()` because a sub-agent's
    `snapshotStore` is the *parent's* (shared via
    `subAgent.snapshotStore = t.mainAgent.snapshotStore`); `Shutdown()` would
    `Reset()` the parent's store and break the TUI sidebar.
- `internal/agent/memory_maintenance.go`
  - Worker now `defer close(a.memoryMaintDone)`.
  - Added `memoryMaintShutdown()` (mirrors `docMaintShutdown`, bounded 5s wait,
    idempotent via `sync.Once`).
- `internal/agent/subagent.go`
  - Synchronous `TaskTool.Execute` branch now `defer subAgent.shutdownTransient()`
    after the sub-agent runs.

Validation done: `go build ./...` ✅, `go vet ./internal/agent/` ✅,
`go test ./internal/agent/ -run 'Discovery|Context|SubAgent|...'` ✅,
`go test -race` on the same set ✅.

## Gap fix: background sub-agents — retain for resume, clean up smartly

### Problem
Background `task` dispatches keep the `Agent` alive after completion because
`run.Sub = subAgent` (`subagent.go:422/448`) holds a reference, and the run
lives in the parent's `AgentRunRegistry`. So the 2 maintenance goroutines leak
for every completed background agent too. The user wants background agents to
**remain available for later resume/inspection**, but not leak goroutines.

### Key insight
A live `Agent` (its goroutines + step loop) is only needed **while running**.
Once a background run reaches a terminal state (`finishOK`/`finishErr`), the
`AgentRun` already holds everything needed for status display and "resume"
(`transcript`, `Result`, `Procs`). `run.Sub` is only consulted for the cosmetic
`ModelLabel()` (`GetProvider()`/`Client().GetModel()` — cheap field reads).
So we can stop the goroutines at completion while keeping the lightweight run
record — that is the "smart" cleanup: **tear down the process, keep the record.**

### Design
1. In the background completion goroutine (`subagent.go`, the `go func()` that
   calls `t.executeSubAgent`), after `run.finishOK(result)` / `run.finishErr(...)`,
   call `subAgent.shutdownTransient()`. This stops the 2 goroutines + closes
   `stopCh`; the `Agent` object and `run.Sub` stay valid (ModelLabel still
   works). This mirrors exactly what B1 does for the synchronous path.
2. **Never** call `shutdownTransient` on a *still-running* background agent
   (it would `Cancel()` and abort the run). It is only invoked after the
   terminal-state transition.
3. **Bounding / reclamation:** rely on the existing `AgentRunRegistry`
   auto-prune (`Snapshot()` → `pruneCompletedLocked(30)`,
   `agent_runs.go:303`). Completed runs beyond the last 30 are dropped from the
   registry; since their goroutines were already stopped at completion, the
   `Agent` (and `run.Sub`) becomes unreferenced and is GC'd. No goroutine leak.
   Running agents are never pruned, so they stay alive for `Cancel`/status.
4. **Parent/session shutdown:** `Agent.Shutdown()` already calls
   `runs.CancelAll()`, which signals `stopCh` on running background agents.
   Their completion handler then fires and runs `shutdownTransient`, so they
   are reaped too. (For a hard process exit the OS reclaims everything.)

### Optional further optimization (lower priority)
After completion, the retained `subAgent` still holds `tools`, `procs`,
`snapshotStore`, `client`, and channels — modest but not free when many
completed runs are retained. To minimize retained footprint:
- Add `Provider`/`Model` strings to `AgentRun` (set at creation from
  `subAgent`), and after `shutdownTransient()` set `run.Sub = nil`.
- This drops the last live `Agent` reference at completion; `ModelLabel()`
  then reads the stored strings instead of `run.Sub`.
Keep as a follow-up; the goroutine leak is fully closed without it.
